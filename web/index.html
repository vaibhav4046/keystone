STEP 1: ISOLATED FILES

Two merge requests touch completely different files. MR-204 edits core/graph.py and MR-211 edits core/audit.py. In Git's eyes, these changes are completely isolated.

GIT CONFLICT: NONE

Git analyzes file paths. Since they edit entirely different files, Git's file-based diff finds zero conflicts. Traditional CI/CD and reviews will greenlight both, risking a broken build in production.

ORBIT SCAN: BLAST RADIUS

Orbit's call graph lets Keystone compute each MR's blast radius. MR-204 (compute_blast_radius) reaches 12 dependents; MR-211 (append) reaches 17. Each count is cross-checked against a live orbit sql query that returns the same number.

orbit sql ring-1 for compute_blast_radius = 12 = engine 12
SILENT COLLISION: BLAST_OVERLAP (severity 6)

Keystone joins the two blast radii and finds they overlap: both MRs ripple into the same 5 dependents. The collision kind and fan-in severity are engine-computed in core/collision.py over the Orbit graph. Git and CODEOWNERS see two unrelated file edits and greenlight both.

shared dependents: approve get_json main post_json precedent

Keystone classifies every cross-MR collision by how the two changes interact on the graph: same_change (both edit one symbol), change_in_blast (one edits inside the other's blast radius), or blast_overlap (their blast radii share dependents, like this one). The full scenario flags two collisions across the three open MRs.

VERDICT: BLOCK (CROSS_TEAM)

Keystone BLOCKs the merge at the CROSS_TEAM tier and returns the safe order from a Kahn topological sort over the collision graph. Merge in any other order and a shared dependent breaks at runtime.
